## Workflow Orchestration

### 1. Plan Mode Default

- Enter plan mode for ANY non-trivial task (3+ steps or architectural decisions)
- If something goes sideways, STOP and re-plan immediately ŌĆö don't keep pushing
- Use plan mode for verification steps, not just building
- Write detailed specs upfront to reduce ambiguity

### 2. Subagent Strategy

- Use subagents liberally to keep main context window clean
- Offload research, exploration, and parallel analysis to subagents
- For complex problems, throw more compute at it via subagents
- One task per subagent for focused execution

### 3. Self-Improvement Loop

- After ANY correction from the user: update `tasks/lessons.md` with the pattern
- Write rules for yourself that prevent the same mistake
- Ruthlessly iterate on these lessons until mistake rate drops
- Review lessons at session start for relevant project

### 4. Verification Before Done

- Never mark a task complete without proving it works
- Ask yourself: "Would a senior developer approve this?"
- Run `npx tsc --noEmit` ŌĆö zero errors before calling anything done
- Check logs, demonstrate correctness

### 5. Demand Elegance (Balanced)

- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution"
- Skip this for simple, obvious fixes ŌĆö don't over-engineer
- Challenge your own work before presenting it

### 6. Autonomous Bug Fixing

- When given a bug report: just fix it ŌĆö don't ask for hand-holding
- Point at logs, errors, failing types ŌĆö then resolve them
- Zero context switching required from the user
- Go fix failing builds without being told how

## Project Overview

Church community management app for FDM with 6 chapters across Metro Manila and nearby provinces.
Handles attendance (QR-based), guest encoding, enthronements, finance, ministries, and announcements.

---

## Tech Stack

- **Framework:** Next.js 15 (App Router, TypeScript)
- **Styling:** Tailwind CSS v4 + shadcn/ui
- **Database:** PostgreSQL via Supabase (Singapore region)
- **ORM:** Prisma 7 ŌĆö client at `lib/generated/prisma/client`
- **Auth:** Supabase Auth ŌĆö passwords stored in `auth.users`, NOT in `public.users`
- **Media:** Cloudinary (no upload presets ŌĆö folder-based via `lib/cloudinary.ts`)
- **Email:** Resend via `lib/email.ts`
- **Hosting:** Vercel

## Database ŌĆö Key Rules

- `public.users` = profile data (Prisma). `auth.users` = credentials (Supabase Auth).
- Link them via `users.auth_id` (Supabase Auth UUID). Null until user activates account.
- Always import `prisma` from `lib/prisma.ts` ŌĆö never instantiate PrismaClient directly.
- Lookup tables use `key` (snake_case machine key) + `name` (human-readable UI label).

## Responsive Typography & Spacing

Always use responsive text sizes ŌĆö never a single fixed size. Follow this scale:

| Use             | Mobile      | Tablet `md:` | Desktop `lg:` |
| --------------- | ----------- | ------------ | ------------- |
| Page title      | `text-xl`   | `text-2xl`   | `text-3xl`    |
| Section heading | `text-lg`   | `text-xl`    | `text-2xl`    |
| Card title      | `text-base` | `text-lg`    | `text-xl`     |
| Body text       | `text-sm`   | `text-base`  | `text-base`   |
| Label / caption | `text-xs`   | `text-sm`    | `text-sm`     |
| Table cell      | `text-xs`   | `text-sm`    | `text-sm`     |
| Button          | `text-sm`   | `text-sm`    | `text-base`   |

Example:

```tsx
<h1 className="text-xl md:text-2xl lg:text-3xl font-bold">Members</h1>
<p className="text-sm md:text-base">Chapter details here</p>
```

**Spacing** ŌĆö always responsive:

```tsx
<div className="p-4 md:p-6 lg:p-8">                      {/* page container */}
<div className="p-3 md:p-4 lg:p-6">                      {/* card */}
<div className="flex flex-col gap-3 md:gap-4 lg:gap-6">  {/* section gap */}
```

**Breakpoints:** `sm` 640px ┬Ę `md` 768px ┬Ę `lg` 1024px ┬Ę `xl` 1280px

**Rules:**

- Always mobile-first ŌĆö base class is mobile, `md:` and `lg:` scale up
- Minimum `text-xs` (12px) ŌĆö never go below
- Body text never below `text-sm` on any screen ŌĆö elderly readability
- Use `font-medium` or `font-semibold` for labels ŌĆö thin fonts are hard to read

## Code Style

- Use TypeScript strictly ŌĆö no `any`
- Server Components by default; add `"use client"` only when needed
- Use `cn()` from `lib/utils.ts` for conditional Tailwind classes
- Prefer `async/await` over `.then()` chains
- All API routes go in `app/api/`
- Always handle loading and error states in UI

## Auth Flow (to be built)

1. Admin creates account from guest log ŌåÆ Supabase Auth invite email sent
2. Member clicks link ŌåÆ first-login page ŌåÆ sets password ŌåÆ `auth_id` saved to `public.users`
3. Session managed via Supabase SSR (`lib/supabase/server.ts`)
4. Route protection via middleware checking Supabase session

## QR Attendance

- Rotating signed JWT per event, 15-min rotation, validated server-side
- Two modes: tablet scanner (Secretariat scans member QR) and self check-in (member scans event QR)
